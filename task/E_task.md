# E_task.md — 成员E任务书：设计报告统稿与最终交付

> **角色定位**：你是这个项目的总装车间。A 做了发动机，B 做了底盘，C 做了车身，D 做了安全系统——你的工作是把它们组装成一辆完整的车，打磨喷漆，然后开上展台。报告到你手上是 4 份风格各异的半成品，从你手上出去是一份统一、整洁、老师愿意给高分的正式文档。

---

## 零、先搞懂：什么叫"统稿"？

### 不是你想象的那样

很多同学以为统稿就是「把 4 个人的 Word 文档拖到一起，加个封面，Ctrl+P」。**大错特错。**

统稿 ≈ 出版社主编的工作：

```
A 交稿                      你拿到手的                      经过你处理后
─────────────────────────────────────────────────────────────────
"这个东西应该这样…"         口头禅、不正式                  "根据需求分析，本系统…"
"我觉得…"                   主观语气                        "经分析，设计方案如下…"
"⻣院系编号 CHAR(4)"          排版乱，字段名不一致           表格统一，格式规范
没有配图说明                  干巴巴的表格                  补充解释文字
各自为政的编号                 1, 2, 3 vs 一、二、三         统一编号体系
A写了4个待确认                 没下文                         追问各组员，补充结论
```

你的工作是让老师翻开报告，**从头到尾读下来感觉是一个人写的**，而不是 5 个人拼凑的。

  4.3  用户自定义完整性（CHECK 约束汇总表）
  4.4  触发器设计（3 个触发器的逻辑说明）

第五章  总结与展望（你自己写，500 字左右）
  5.1  设计总结（本项目解决了什么问题）

## 四、正式步骤

### 第 1 步：提前收集素材（贯穿第 3-6 天）

不要等到所有人都交了才开始干活。**在第 3 天就开始动手**：

| 时间 | 行动 |
|------|------|
| **第 3 天** | A 交需求说明书 → 你当天就复制到报告模板里，调好第一章的格式，发现缺失当场问 A |
| **第 4 天** | B 交 E-R 图 → 你当天插入报告第二章。检查图片是否清晰、标注是否可读 |
| **第 5 天** | C 交关系模式 → 你当天排第三章的 18 张表。检查字段名和 B 的图是否一致 |
| **第 6 天** | D 交约束 → 你当天排第四章。检查外键名和 C 的表是否对应 |
| **第 6-7 天** | 全程穿插：写第 5 章总结、调排版、补过渡段、追问组员模糊的地方 |

> 💡 核心策略：**收到一份，排一份，问一份**。不要让问题堆到最后一天。

---

### 第 2 步：统一格式规范（第 6 天集中做，1 小时）

这是区分「能看」和「专业」的关键。给全文立一套规矩并严格执行：

| 规则项 | 统一要求 |
|--------|---------|
| **字体** | 正文宋体/小四，标题黑体，英文/代码用 Consolas 或 Courier New |
| **表格样式** | 三线表（顶线、底线粗，中间线细），表头加底灰 |
| **图片** | 全部矢量图（SVG/PNG 高清），居中，图下方标注"图 X-X  xxx" |
| **编号** | 章→节→小节 统一用 1 / 1.1 / 1.1.1 |
| **术语** | 全篇统一：主键就叫"主键"不要一会儿叫"主码"一会儿叫"PK" |
| **表格表头** | 字段名/类型/约束/说明 四列统一，不要A的四列、C的五列、D的六列 |
| **外键写法** | 统一用 `FK → TableName(field)` |
| **页码** | 封面无页码，目录用罗马数字，正文用阿拉伯数字 |

---

### 第 3 步：补过渡和衔接（第 7 天，1-2 小时）

4 个人写的 4 章之间很可能是断裂的——第 1 章结尾和第 2 章开头没有半点关系。你要补上**过渡段**：

**示例——第一章末补一句**：
> "基于以上需求分析，下面采用 E-R 模型进行概念结构设计，将 11 个业务模块抽象为 18 个实体及其联系。"

**示例——第二章末补一句**：
> "E-R 图描述的是概念层的数据模型，与具体数据库无关。下面将其转换为 MySQL 支持的关系模式。"

**示例——每张表格前补一句引导**：
> "表 3-1 列出了 STUDENT 表的关系模式，其中 stu_id 为主键，dept_id 和 major_id 为外键。"

> 这些过渡句看似废话，但没有它们，报告读起来就像四份独立文档被订书机钉在一起。老师一看就知道谁偷懒了。

---

### 第 4 步：一致性审查（第 7 天，1-2 小时）⚠️ 最重要

这是统稿人最核心的工作，也是最容易被忽视的。

**检查清单**：

#### 4.1 术语一致性

全文搜索以下词，确保只有一种写法：

```
"主键"     vs "主码"      vs "PK"      → 统一为 "主键 (PK)"
"外键"     vs "外码"      vs "FK"      → 统一为 "外键 (FK)"
"学号"     vs "学生编号"   vs "stu_id"  → 在中文正文用"学号"，在表格用"stu_id"
"教学班"   vs "开课班"     vs "班级"    → 统一为 "教学班"
```

#### 4.2 数据一致性

| 检查项 | 怎么做 |
|--------|--------|
| A 的 11 个模块名称 = B 的实体名称 = C 的 18 张表名？ | 交叉比对，不一致的找对应组员确认 |
| B 的 E-R 图中的字段名 = C 表中关系模式的字段名？ | 抽样 5 张表字段比对 |
| C 的外键列表 = D 的外键约束列表？ | 都应该是 28 条，数量对上 |
| 18 张表在报告里出现了 18 次（不多不少）？ | 在第三章数一遍 |

#### 4.3 图表编号一致性

```
文中引用              实际图表编号
"如图 2-1 所示"   →  图下方标注是否为 "图 2-1"？
"见表 3-2"        →  表上方标注是否为 "表 3-2"？
```

> Word 里用"引用 → 交叉引用"功能可以自动维护编号，推荐使用。

### 第 5 步：写总结章节（第 7 天，1 小时）

第五章自己写，控制在 500 字左右。三段式：

```markdown
## 5.1 设计总结

本项目设计了一个包含 18 张表的教务管理数据库，覆盖学生、教师、课程、
选课、排课、成绩、考勤、调课、毕业设计、项目设计等 11 个业务模块。
核心设计亮点包括：
- 通过 COURSE_LEVEL_RULE 表解决了本/研课程学分差异化问题
- 通过 ENROLLMENT 中间表拆解了学生与课程的多对多关系
- 通过 GRADE_ITEM + GRADING_TEMPLATE 支持了成绩组成的灵活自定义
- 通过 3 个触发器实现了选课容量、时间冲突、权重总和等动态约束

## 5.2 不足与改进

- 教室冲突检测的触发器实现较复杂，当前采用伪代码描述，具体实现留待后续
- 未涉及用户权限管理模块，实际系统需补充
- 并发选课的事务处理未在数据库层完全展开
- 毕业设计和项目设计的评分流程可以更细化

## 5.3 小组协作体会

本次项目中，小组 5 人分工明确（需求分析→E-R图→关系模式→完整性约束→报告统稿），
采用流水线协作模式，通过 AGENTS.md 和需求分析说明书建立了统一的数据字典，
各成员的作品实现了较高的耦合度和一致性。（这里根据实际情况补充真实体会）
```

### 第 6 步：排版润色（第 7 天，1 小时）

- 生成自动目录（Word: 引用 → 目录）

- 检查封面信息是否完整（5 人姓名学号一个不漏）
- 导出 PDF，在不同设备上打开看看格式有没有乱

## 五、常见错误（提前避开）

| ❌ 错误 | ✅ 正确做法 |
|---------|-----------|
| 把 4 个人的文档直接粘贴，字体字号各不一样 | 粘贴后用"格式刷"或"清除格式"重新统一排版 |
| 正文用"我觉得""这里应该"等口语 | 全局替换为客观语气 |
| 表格截图而不是真正的表格 | 表格必须是可编辑的 Word 表格，不能是图片 |
| E-R 图模糊看不清标注 | 找 B 要 SVG/PDF 矢量版，或放大导出高清 PNG |
| 目录是手打的 | 用 Word 自动目录功能 |
| 总结写"本次项目收获很大"就没了 | 至少写 500 字，含设计亮点、不足、协作体会 |
| 忘了写小组分工 | 封面或前言里必须注明每人负责什么 |
| 附录里漏了东西 | 对照三（二）的交付物清单，附录里应该收齐全套 |

## 八、催稿话术（实战工具）

作为统稿人，你天然有权催促前面的人。以下是各阶段的标准话术：

| 场景 | 话术 |
|------|------|
| 第 3 天 A 还没交 | "A，我已经建好报告模板了，你的需求分析写好了吗？我先排第一章~" |
| 第 4 天 B 还没交 | "B，E-R 图画好了吗？C 等着你的图开始写关系模式，我也需要排第二章了" |
| 第 5 天 C 还没交 | "C，进度怎么样？D 等你交了他才能开始写约束，发我个半成品也行我先排着" |
| 第 6 天 D 还没交 | "D，我第四章留好位置了，你随时发我随时填。今晚之前能给我吗？" |
| 发现不一致 | "B 你的 student 表里有个字段叫 sex，但 A 文档里叫 gender，确认一下用哪个？" |

> 核心技巧：不要质问"你怎么还没交"，而是说"我已经准备好了，随时可以接你的产出"。后者给别人压力但不让人反感。
