## 零、先搞懂：E-R 图到底是什么？

### 用大白话说

E-R 图就是「实体-关系图」——把现实世界里的东西和它们之间的联系画出来。就像画一张地图：

| 地图概念 | E-R 图对应 | 举例 |
|----------|-----------|------|
| 城市 | **实体**（Entity） | 学生、课程、教师 |
| 城市的人口、面积 | **属性**（Attribute） | 学生的学号、姓名、年龄 |
| 城市之间的公路 | **联系**（Relationship） | 学生 **选** 课程 |
| 一条路能跑几辆车 | **基数**（Cardinality） | 一个学生选多门课，一门课被多人选 |

一张好的 E-R 图，任何人看一眼就应该能说出："这个系统里有哪些数据？它们之间怎么连接的？"

---

### E-R 图的 4 个核心元素

```
┌──────────────┐           ┌──────────────┐
│    实体       │           │    属性       │
│  ┌────────┐  │           │  年龄 ──┐     │
│  │ 学生   │  │           │  姓名 ──┤     │
│  └────────┘  │           │  学号 ──┘     │
│   矩形表示    │           │  椭圆表示     │
└──────────────┘           └──────────────┘

       联系（菱形）                    基数标注（线上）
  学生 ────选课──── 课程          学生 1 ──────── N  课程
        │                              │              │
        │                              │  一个学生    │
        │                              │  选 N 门课   │
```

**但是**，现在教学和实际作业中更常用的是一种简化画法——**直接用矩形表示属性（像数据库表一样）**，而不是画一堆椭圆。两种都可以，和 C/D 统一一下风格就行。下文默认用「矩形框+字段列表」的方式，因为 C 可以直接照着建表。

### 第 3 步：画联系（核心工作，2-3 小时）

这是 E-R 图**最核心也最容易出错**的部分。

#### 3.1 联系怎么画？

联系用**菱形**（或一条带标签的线）：

```
┌──────────┐         ┌──────────┐
│ STUDENT  │───选课──▶│ENROLLMENT│◀───包含───┐
└──────────┘   (1:N)  └──────────┘    (1:N)  │
                                       ┌─────┴──────┐
                                       │COURSE_     │
                                       │OFFERING    │
                                       └────────────┘
```

> 如果工具不支持菱形，用直线上标文字也行：`STUDENT ——"选课(1:N)"——▶ ENROLLMENT`

#### 3.2 联系上的基数怎么写？

标记在线的两端，靠近实体的一侧：

```
    1                 N
STUDENT ───────选课─────── ENROLLMENT

读法：1 个学生 → N 条选课记录
      N 条选课记录 → 1 个学生
```

| 符号 | 含义 | 例子 |
|:----:|------|------|
| 1 | 有且只有一个 | 一个学生只有一个毕设 |
| N (或 M) | 可以有多个 | 一个教师指导多个毕设 |
| (1,1) | 强制且唯一 | 选课必须对应一个学生 |
| (0,N) | 可选多个 | 学生可以没有项目 |

#### 3.3 完整联系清单（从 A 的 §6.5 抄过来）

对照 §6.5 的 30 条关系，一条一条画。下面按优先级排序，先画核心的：

**第一优先级（必须画对的核心联系）**：

```
STUDENT  1──N  ENROLLMENT  N──1  COURSE_OFFERING  N──1  COURSE
                                        │
                                        N──1  TEACHER
                                        │
                                        1──N  SCHEDULE  N──1  CLASSROOM
                                        │
                                        N──1  SEMESTER
```

> 这段是系统的脊椎：学期里教师开课 → 学生选课 → 排时间教室。画对了这段，其他都是旁支。

**第二优先级（成绩、考勤、调课）**：

```
STUDENT  1──N  GRADE  N──1  COURSE_OFFERING

GRADE    1──N  GRADE_ITEM

SCHEDULE  1──N  ATTENDANCE  N──1  STUDENT

SCHEDULE  1──N  SCHEDULE_CHANGE  N──1  TEACHER

CLASSROOM  1──N  SCHEDULE_CHANGE
```

**第三优先级（毕设、项目、学分规则）**：

```
STUDENT  1──1  THESIS  N──1  TEACHER (指导)
                      N──1  TEACHER (评阅)

STUDENT  1──N  PROJECT  N──1  TEACHER
                │
                N──1  COURSE

COURSE  1──N  COURSE_LEVEL_RULE

DEPARTMENT  1──N  (STUDENT, TEACHER, COURSE, MAJOR)

MAJOR  1──N  STUDENT
```

### 第 4 步：处理特殊情况（1 小时）

#### 4.1 自引用

COURSE 表里有一个 `prereq_id`（先修课程），它指向自己。这种在 E-R 图上画成：

```
        先修课程 (1:N)
    ┌──────────────────┐
    ▼                  │
┌────────┐            ┌─┴──────┐
│ COURSE │            │ COURSE │
│ (后修) │            │ (先修) │
└────────┘            └────────┘
```

#### 4.2 同一个实体参与两种联系

TEACHER 和 THESIS 之间有两条线：一条是「指导」，一条是「评阅」。这是两个不同的联系，要画两条线，标不同的联系名。

```
           指导 (1:N)
TEACHER ────────────────▶ THESIS
        评阅 (1:N)
      ────────────────▶
```

#### 4.3 可选联系

有些联系不是强制的，比如学生不一定要有毕设（大一大二没有）。画图时标注：

```
STUDENT ────────毕设(0..1:1)──────── THESIS
```

### 第 6 步：写设计说明（1 小时）

E-R 图不是哑图，要附文字解释。写一份 `E-R图设计说明.md`：

```markdown
# E-R 图设计说明

## 1. 实体设计依据
- 为什么拆了 18 个实体？（从 A 的 11 个模块扩展而来）
- 哪些是需求里没有但你自己加的？（如 Department, Major, Semester）
- 哪些是中间表/关联实体？（Enrollment 拆解多对多）

## 2. 关键联系说明
- 选课为什么三张表？（Student → Enrollment → CourseOffering，拆多对多）
- 成绩为什么要拆 GRADE + GRADE_ITEM 两张表？
- 调课为什么指向 SCHEDULE 而不是 COURSE_OFFERING？

## 3. 基数设计理由
- 为什么 THESIS 和 STUDENT 是 1:1？
- 为什么 COURSE 和 COURSE_OFFERING 是 1:N？
- 为什么 PROJECT.stu_id 允许 N:N（一个学生多个项目，一个项目多个学生）？

## 4. 设计取舍
- 教室特征为什么只保留容量？
- 为什么没有把「上课班级」单独做实体？
```

## 四、常见错误（提前避开）

| ❌ 错误 | ✅ 正确做法 |
|---------|-----------|
| 把 ENROLLMENT 的属性塞进 STUDENT 里 | ENROLLMENT 独立成表，别偷懒 |
| 实体之间直接连线不标基数 | 每条线两端都要标 1 或 N |
| 用一根线连接 COURSE_OFFERING 到所有表 | 从 COURSE_OFFERING 出发，分别连到不同子图 |
| 忽略自引用（课程→先修课程） | COOURSE 画一条回到自己的弧线 |
| E-R 图上直接写 SQL 外键约束 | E-R 图画概念联系，C 负责转成外键 |
| 属性表里漏掉类型 | 每个属性旁边标注 CHAR(10) / INT / DATE 等 |
| 18 个实体全挤一张图 | 至少拆 3 张分图 |

## 七、和 C 的对接

你是 C 的直接上游。C 能写成什么样，取决于你画成什么样：

1. **你的实体名 → C 的表名**（保持一致）
2. **你的属性名 → C 的字段名**（保持一致）
3. **你的联系 → C 的外键**（你的 N:1 线 = C 的外键方向）
4. **你的基数 → C 的约束**（你的 1:1 = C 的 UNIQUE，你的 1:N = C 的普通 FK）

> 💡 建议画完第一版后，马上给 C 看 5 分钟——C 能指出哪里不清楚，趁早改比交付后返工强 10 倍。
