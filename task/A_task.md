# A_task.md — 成员A任务书：需求分析

> **角色定位**：你在5人小组中负责第一阶段「需求分析」，是项目的起点。你的产出是 B（E-R图）和 C（关系模式）的直接输入，质量决定整个项目的天花板。

---

## 一、你要做什么

把学校教务管理系统的业务需求从模糊的一句话变成 **精确到字段级别的结构化描述**，让后续同学不需要再猜"这个表到底要存什么"。

**一句话交付标准**：B拿着你的文档，能直接画出 E-R 图；C拿着你的文档，能直接写出 CREATE TABLE 语句。

---

## 二、具体步骤（按顺序做）

### 第1步：通读原始需求（30分钟）

打开原始文档，对照下面的清单逐项理解：

- [ ] 11 个模块分别是什么业务场景？
- [ ] 每个模块涉及哪些"人"和"东西"？
- [ ] 哪些模块之间有明显的关联（如选课连接学生和课程）？
- [ ] 11 条"问题说明"每条问的是什么设计取舍？

---

### 第2步：逐个拆解实体属性（核心工作，3-4小时）

对每个模块，列出它需要的所有属性字段。每个字段必须回答3个问题：

| 必须回答 | 示例 |
|----------|------|
| **叫什么？** | `stu_name`（姓名） |
| **什么类型？** | 字符串、整数、日期、布尔... |
| **有什么约束？** | 非空？唯一？默认值？取值范围？外键指向谁？ |

**参考模板**（以「学生基本信息」为例）：

```
实体：学生 STUDENT
┌──────────┬──────────┬──────────┬──────────┬────────────────┐
│ 字段名    │ 含义      │ 类型      │ 必填?    │ 约束/说明       │
├──────────┼──────────┼──────────┼──────────┼────────────────┤
│ stu_id   │ 学号      │ CHAR(10) │ ✓        │ PK，如2024010001│
│ stu_name │ 姓名      │ VARCHAR  │ ✓        │                │
│ gender   │ 性别      │ CHAR(1)  │          │ CHECK 男/女     │
│ degree   │ 学历层次  │ CHAR(4)  │ ✓        │ 本科/硕士/博士   │
│ dept_id  │ 院系编号  │ CHAR(4)  │          │ FK→Department   │
│ ...      │ ...      │ ...      │ ...      │ ...            │
└──────────┴──────────┴──────────┴──────────┴────────────────┘
```

> ⚠️ **常见错误提醒**：
> - 不要把"上课班级"当成学生的一个字段——它随选课动态产生，不属于学生固定属性
> - 不要把"成绩"塞进学生表——成绩是学生+课程的关系，应独立成表
> - 类型要具体：不要写"字符串"，要写 CHAR(10) 还是 VARCHAR(50)

---

### 第3步：梳理实体间的关系（1-2小时）

画出**初步关联草图**（纸笔即可，不需正式E-R图）：

```
学生 ──(选课)──▶ 开课 ◀──(授课)── 教师
                    │
                    ├──(排课)──▶ 教室
                    │
                    └──(成绩)──▶ 成绩 ──(子项)──▶ 成绩子项
```

对每个关系回答：
- **基数**：一对一？一对多？多对多？（如：一个学生选多门课、一门课被多个学生选 → 多对多，需要中间表 Enrollment）
- **是否强制**：学生可以没有毕设吗？（可以，大一大二没有）→ 非强制关系

---

### 第4步：处理设计问题（1-2小时）

原始文档有11条可选问题，逐条给出**明确决策**（不要写"可A可B"，必须选一个）：

| # | 问题 | 你需要回答 |
|---|------|-----------|
| 1 | 学分制：选修/必修如何定？ | 在哪张表里区分？按什么规则？ |
| 2 | 本研课程重叠 | 同一门课本研不同性质怎么存？ |
| 3 | 管理班级 vs 上课班级 | 关联还是不关联？怎么设计？ |
| 4 | 人数限制 | 上限存哪里？选课时怎么校验？ |
| 5 | 分班 | 同一课程多个班怎么表示？ |
| 6 | 不规则上课时间 | 隔周、前8周/后8周怎么存？ |
| 7 | 实验课时间不确定 | 允许为空还是必须有默认值？ |
| 8 | 实践项目独立 | 独立项目怎么关联？ |
| 9 | 评分标准统一？ | 百分制还是多标准？ |
| 10 | 成绩自定义组成 | 子项权重怎么存？ |
| 11 | 教室特征 | 哪些特征纳入数据库？ |

每条决策要写清楚 **"存在哪个表、用哪个字段"**，不要只写概念。

---

### 第5步：写非功能需求（30分钟）

至少覆盖：

- 数据完整性（外键约束、非空约束）
- 并发场景（选课抢课）
- 历史数据保留（学生毕业后）
- 操作追溯（谁在什么时候改了什么）

---

### 第6步：列出待确认事项（30分钟）

把你拿不准的、或者觉得需要全组讨论的点列出来，格式：

> - [ ] 是否需要单独的院系表？还是把院系名直接存在学生表里？
> - [ ] 是否需要学期表统一管理？
> - [ ] 考勤是否和平时成绩自动挂钩？
