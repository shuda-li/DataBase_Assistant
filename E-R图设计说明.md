# E-R 图设计说明

> 配合《教务管理系统 E-R 图》阅读。解释 18 个实体、30 条关系、每条基数的设计理由。

---

## 一、实体设计依据

### 1.1 为什么是 18 个实体？

A 的需求分析说明书覆盖了 **11 个业务模块**，但直接按模块建表会导致大量冗余（比如"学生"模块里塞选课信息、"课程"模块里塞开课信息）。在设计 E-R 图时，我对 11 个模块做了以下扩展：

| 来源 | 实体 | 说明 |
|------|------|------|
| §2.1 → 扩展 | STUDENT | 学生基本信息，独立实体 |
| §2.2 → 扩展 | TEACHER | 教师基本信息，独立实体 |
| §2.3 → 扩展 | COURSE | 课程目录，独立实体 |
| §2.3 → **新增** | COURSE_LEVEL_RULE | §3.2 需求中提到本/硕/博同一门课学分不同、性质不同，必须拆成独立规则表 |
| §2.4 → 扩展 | COURSE_OFFERING | 开课（教学班），拆出来是因为一门课多个学期多个教师开多个班 |
| §2.5 → **新增** | ENROLLMENT | 学生和开课之间是多对多（一个学生选多门课，一门课被多人选），必须用中间表拆 |
| §2.6.1 → 扩展 | CLASSROOM | 教室独立实体（被排课和调课引用） |
| §2.6.2 → 扩展 | SCHEDULE | 排课记录，独立实体 |
| §2.7.1 → 扩展 | GRADE | 成绩主表，独立实体 |
| §2.7.2 → **新增** | GRADE_ITEM | 拆出来因为一门课的成绩由多个子项组成（平时/期中/期末/实验），一对多 |
| §2.7.3 → **新增** | GRADING_TEMPLATE | 评分模板，预设权重，避免逐生重复录入 |
| §2.8 → 扩展 | THESIS | 毕业设计，与课程成绩体系独立 |
| §2.9 → 扩展 | PROJECT | 项目设计，允许独立于课程的实践项目 |
| §2.10 → 扩展 | ATTENDANCE | 考勤，与排课绑定 |
| §2.11 → 扩展 | SCHEDULE_CHANGE | 调课记录，指向排课而非开课 |
| §5 待确认 → **新增** | DEPARTMENT | 院系表，减少学生/教师/课程中的院系名称冗余 |
| §5 待确认 → **新增** | MAJOR | 专业表，统一管理专业信息 |
| §5 待确认 → **新增** | SEMESTER | 学期表，统一学期编号，便于按学期查询 |

**结论**：11 个模块 → 18 个实体，其中 5 个是新增的基础字典或中间表，都是为了消除冗余、保证数据一致性。

### 1.2 哪些是"关联实体"（中间表）？

以下实体的本质是**拆解多对多关系**：

- **ENROLLMENT**：STUDENT（学生）↔ COURSE_OFFERING（开课）是多对多 → 用 ENROLLMENT 拆成两个一对多
- **GRADE_ITEM**：GRADE（成绩）↔ 子项 是一对多 → 拆出去可以灵活增删子项
- **COURSE_LEVEL_RULE**：COURSE（课程）↔ 学历层次+性质+学分 是一对多 → 一门课在不同学历下有不同规则

## 二、关键联系说明

### 2.1 选课为什么是三张表？

```
STUDENT ——1:N—— ENROLLMENT ——N:1—— COURSE_OFFERING ——N:1—— COURSE
```

而不是 `STUDENT ——N:M—— COURSE_OFFERING` 直接连：

- 如果直接连，选课时间、选课状态、课程性质标记这些信息没地方存（它们是选课行为本身的属性，不属于学生也不属于开课）
- ENROLLMENT 作为中间表，承担了"选课记录"的独立属性：enroll_time（选课时间）、enroll_status（已确认/已退选）、real_nature（学生实际是必修还是选修）
- 这也方便后续查询——直接查 ENROLLMENT 表就能知道"某学生这学期选了什么课、什么时候选的、状态如何"

### 2.2 成绩为什么拆成 GRADE + GRADE_ITEM 两张表？

需求 §3.6 明确说"成绩由多个部分组成，组成形式可支持自定义"。如果只建一张 GRADE 表，字段会变成"平时成绩""期中成绩""期末成绩""实验成绩"……不同课程子项不同，有的有实验分有的没有，列会大量空值。

拆法：
- **GRADE**：只存总评成绩、等级、是否补考等"成绩级"信息
- **GRADE_ITEM**：每个子项一条记录（平时/期中/期末/实验……），每条有独立的得分和权重
- 总评成绩 = Σ(子项得分 × 权重)，可在应用层计算后写回 GRADE.total_score

### 2.3 调课为什么指向 SCHEDULE 而不是 COURSE_OFFERING？

因为调的是"一次具体的排课安排"，不是"整个教学班"：

- 某教学班（COURSE_OFFERING）一周可能有 2 次课：周一 1-2 节 + 周三 3-4 节
- 老师只想调周三那次 → 必须指向具体的 SCHEDULE 记录
- 如果指向 COURSE_OFFERING，系统不知道调的是哪一次

所以：SCHEDULE_CHANGE.sch_id → SCHEDULE（而非 COURSE_OFFERING）

### 2.4 COURSE 自引用（先修课程）

COURSE.pre_course_id 指向 COURSE.course_id 自身，形成一条回到自己的弧线。例如：
- "数据结构（CS21003）"要求先修"C 语言程序设计（CS21001）"
- 在 COURSE 表中，CS21003 的 pre_course_id = "CS21001"

E-R 图上画成 `COURSE ——1:N（先修课程）——▶ COURSE` 的自引用环。一个课程只有一个先修课程，但一个课程可以成为多个后续课程的先修课，所以是 1:N。

## 三、基数设计理由

### 3.1 为什么 THESIS 和 STUDENT 是 1:1？

- **业务逻辑**：一个学生只有一个毕业设计（本科生一个论文题目，研究生一个学位论文）
- **反过来**：一个毕设只属于一个学生（不存在几个学生合写一个毕设的情况）
- 注意：大一大二学生没毕设，不会出现在 THESIS 表中。出现在表里的学生必然有且仅有一条毕设记录

因此标注：`STUDENT ||--|| THESIS`（1:1）

### 3.2 为什么 COURSE 和 COURSE_OFFERING 是 1:N？

- 一门课程（如"高等数学"）可以在**不同学期**由**不同教师**开设**多个教学班**
- 例如：2024-2025-1 学期，高等数学开了 5 个班（张三教 1-2 班，李四教 3-5 班）
- 每个教学班（COURSE_OFFERING）只对应一门课程

因此：`COURSE ||--o{ COURSE_OFFERING`（1:N）

### 3.3 为什么 STUDENT 和 PROJECT 是 1:N？

- 一个学生可以参与多个项目（比如同时做"数据结构课程设计"和一个"独立实践项目"）
- 反过来，PROJECT.stu_id 是一个外键字段，允许多个学生参与同一个项目（但每条记录只关联一个学生——如果一个项目多个学生参与，就插入多条 PROJECT 记录）

这里标注为 1:N，表示一个学生可以有 N 条项目记录。

### 3.4 为什么 TEACHER → THESIS 有两条线？

因为教师和毕设之间存在**两个不同的角色**：

- **指导教师（advisor）**：负责指导这篇论文，THESIS.advisor_id → TEACHER.teacher_id
- **评阅教师（reviewer）**：负责评审这篇论文，THESIS.review_teacher_id → TEACHER.teacher_id

这是两个不同的联系（不同的语义、不同的 FK 字段），必须在 E-R 图上画两条独立的线并分别标注"指导"和"评阅"。同一个教师可以既指导某些毕设又评阅另一些毕设。

## 四、设计取舍

### 4.1 教室特征为什么只保留容量、类型、投影仪、桌椅可移动？

需求 §3.7 的最终决策是"所有教室基本设施齐全，只需要根据选课人数匹配容量；机房类课程强绑定即可"。因此：

- 保留 `capacity`：排课时做容量校验（选课人数 ≤ 教室容量）
- 保留 `room_type`：区分普通教室/实验室/多媒体教室，供排课匹配
- 保留 `has_projector` / `projector_num` / `desk_movable`：供精细排课使用
- 不保留楼层、朝向等物理属性：对排课逻辑无实际约束作用

### 4.2 为什么没有把"上课班级"单独做成实体？

需求 §3.1 已有明确决策：**管理班级 ≠ 上课班级**。上课班级是选了同一个教学班的学生集合，通过 `ENROLLMENT` 表 JOIN `COURSE_OFFERING` 即可动态算出，不需要单独建表。单独建"上课班级表"反而会带来同步维护问题——每次有人退选就得改。

### 4.3 为什么"课程性质"不直接存在 COURSE 表里？

需求 §3.2 指出：同一门课程（如"高等数学"）对本科生是必修、对硕士生是选修、学分也不同。如果把 course_nature 放在 COURSE 表里，只能存一个值，无法区分不同学历。

解决方案：设计 COURSE_LEVEL_RULE 表，按（课程编号 + 学历层次）组合来确定性质和学分。选课时根据学生的 edu_level 自动匹配。

### 4.4 STUDENT 里既有 dept_id 又有 major_name？

A 文档 §2.1 的属性列表中，"院系编号"是 FK，"专业名称"是 VARCHAR。这个设计是有意的——dept_id 指向 DEPARTMENT 表保证参照完整性，major_name 直接存专业名称（冗余但查询快）。MAJOR 表作为独立字典存在，供下拉选择等前端使用。两者并不矛盾。

## 五、与 C 的对接要点

| E-R 图里的… | C 写 SQL 时对应… |
|-------------|-----------------|
| 实体名（如 STUDENT） | `CREATE TABLE STUDENT` |
| 属性名（如 stu_id） | 列名 `stu_id CHAR(10)` |
| PK 标记 | `PRIMARY KEY (stu_id)` |
| FK 标记（如 dept_id FK） | `FOREIGN KEY (dept_id) REFERENCES DEPARTMENT(dept_id)` |
| 1:N 连线（方向 N 端→1 端） | 在 N 端加 FK 指向 1 端的 PK |
| 1:1 连线（STUDENT↔THESIS） | THESIS.stu_id 加 UNIQUE 约束 |
| 自引用（COURSE→COURSE） | `FOREIGN KEY (pre_course_id) REFERENCES COURSE(course_id)` |
| 并行联系（TEACHER→THESIS×2） | THESIS 表里建两个 FK 字段：advisor_id 和 review_teacher_id |
